## Context

泰迪杯 B 题，中药上市公司财报"智能问数"助手。示例数据：华润三九（深交所 000999）、金花股份（上交所 600080），39 份财报 PDF，2022-2025 年报/季报/半年报。

赛题三个任务：任务一 PDF→DB、任务二 NL2SQL、任务三 RAG。Phase 1 覆盖任务一全部 + 任务二核心。

关键约束：
- Schema 由官方指定（附件3），4 张表，字段固定
- report_period 格式："2025Q3"/"2022FY"/"2024HY"/"2023Q1"
- 4.25 测试给新数据，6 小时内 Pipeline 跑完出结果
- 输出必须按附件7 JSON 格式 + xlsx

## Goals / Non-Goals

**Goals:**
- 华润三九 + 金花股份全部 PDF 自动解析入库，4 张表数据完整
- 数据校验通过（勾稽关系、跨表一致性）
- 附件4 的 2 个示例问题能正确回答
- `python pipeline.py --task etl` 一键入库
- 输出符合附件7 格式的 result_2.xlsx + 图表图片

**Non-Goals:**
- 研报 RAG / 归因分析（Phase 3，任务三）
- OpenViking 记忆系统（Phase 3，P2 创新点）
- 多意图拆解（Phase 3）
- UI 美化（Phase 4）

## Decisions

### 1. PDF 表格提取：以规则抽取为主，暂不把 LLM fallback 设为 Phase 1 主路径

**修正后的选择**：pdfplumber 负责文本与表格抽取，字段归一化采用规则映射、别名词典、标题识别和跨页拼接；LLM 仅作为离线分析或后续增强，不应作为 Phase 1 的默认在线兜底路径。

**实测依据**：
- 华润三九 2023 年年报：
  - 第 7-8 页可直接抽到“主要会计数据和财务指标”“分季度主要财务指标”表。
  - 第 100 页可直接抽到“合并资产负债表”。
  - 第 103 页可直接抽到“合并利润表”。
  - 第 106 页可直接抽到“合并现金流量表”。
- 金花股份 2023 年年报：
  - 第 6 页可直接抽到“近三年主要会计数据和财务指标”。
  - 第 80 页可直接抽到“合并资产负债表”。
  - 第 84 页可直接抽到“合并利润表”。
  - 第 87 页可直接抽到“合并现金流量表”。

**为什么要修正**：
- 示例 PDF 中核心表已能被 pdfplumber 直接抽出，主链路具备规则实现可行性。
- LLM fallback 会引入额外延迟、成本和不可复现性，不利于 4.25 新数据 6 小时全自动跑完。
- 任务一更强调可靠、可审计、可批处理，Phase 1 应优先 deterministic pipeline。

**实现建议**：
1. 第一阶段仅实现规则映射：`标准中文行项目 -> 附件3字段`。
2. 对常见别名做静态归并，如“营业总收入/营业收入”“销售商品、提供劳务收到的现金”等。
3. 未命中字段先记录 warning 和 unmapped rows，不阻塞整体表落库。
4. 如后续发现规则覆盖不足，再增加可开关的 LLM 建议模式，但不作为主流程强依赖。

### 2. 上交所 / 深交所 双格式处理

**选择**：根据目录名或文件名模式自动判断交易所，分别解析。

**深交所**：`华润三九：2023年年度报告.pdf` → 文件名直接解析出公司名 + 年份 + 报告类型。

**上交所**：`600080_20240427_0WKP.pdf` → `stock_code` 从文件名取；`report_period` 必须优先从 PDF 首页标题推断，发布日期只作为弱兜底，不能单独作为最终判定依据。

**为什么要修正**：
实测同一发布日期下，上交所有多份不同类型文件：
- `20230428` 同时存在：2022 年年度报告、2022 年年度报告摘要、2023 年第一季度报告
- `20240427` 同时存在：2023 年年度报告、2023 年年度报告摘要、2024 年第一季度报告
- `20250425` 同时存在：2024 年年度报告、2024 年年度报告摘要、2025 年第一季度报告
- `20240817` / `20250822` 同时存在：半年度报告、半年度报告摘要

因此，原设计中的“按发布日期推断报告期”只能作为候选规则，**无法解决同日多份 PDF 的报告类型识别问题**。

**实现建议**：
- 首页标题优先级：
  1. `XXXX年年度报告`
  2. `XXXX年年度报告摘要`
  3. `XXXX年第一季度报告`
  4. `XXXX年半年度报告`
  5. `XXXX年半年度报告摘要`
  6. `XXXX年第三季度报告`
- 只对“完整版报告”入库，摘要文件默认跳过。
- `report_period` 由标题直接映射：年度→`{year}FY`，第一季度→`{year}Q1`，半年度→`{year}HY`，第三季度→`{year}Q3`。
- 若首页标题缺失，再退回发布日期 + 全文关键字联合判断。

### 3. 数据库：SQLite，Schema 对齐官方

**选择**：SQLite，表名和字段完全对齐附件3（`core_performance_indicators_sheet`, `balance_sheet`, `income_sheet`, `cash_flow_sheet`）。

**替代方案**：MySQL（赛题附件3 提到 MySQL 字段类型）。

**理由**：SQLite 零配置、轻量；SQL 语法与 MySQL 高度兼容，表结构可保持一致。如果评审要求 MySQL，切换主要发生在连接层。

**补充修正**：需要严格承认附件3中的字段单位并不统一，不能简单概括成“官方 Schema 要求万元”。

**实测依据**：
- 大部分财务金额字段是“万元”
- `cash_flow_sheet.net_cash_flow` 明确是 **元**
- EPS、每股净资产、每股经营现金流量等字段也是“元”
- 百分比字段是 `%`

因此不能做“除每股指标外全部 ÷10000”的粗粒度规则，必须按字段级别定义单位转换策略。

### 4. 字段映射策略

**修正后的选择**：维护“表内行项目中文名 -> 官方字段名”的静态映射字典，并为每个字段定义单位策略；不假设 PDF 指标名与附件3 一一同名，也不假设 4 张表都能完全由单张原始报表直接填满。

ni**实测结论**：
- `balance_sheet`、`income_sheet`、`cash_flow_sheet` 的大部分字段都能从“合并资产负债表 / 合并利润表 / 合并现金流量表”直接获得。
- `core_performance_indicators_sheet` 更像“主要会计数据和财务指标”汇总表，其字段需要从“主要会计数据”“主要财务指标”“分季度主要财务指标”等章节抽取，和三大报表不是同一个来源。
- 附件3中的部分字段是衍生指标或同比/环比指标，需要二次计算，不能只靠报表原值映射。

**落地策略建议**：
- `balance_sheet`：优先取“合并资产负债表”本期末值；同比字段通过上年期末值计算。
- `income_sheet`：优先取“合并利润表”本期值；同比字段由上年同期计算。
- `cash_flow_sheet`：优先取“合并现金流量表”；`net_cash_flow` 保持元，其他注明万元的字段再转换。
- `core_performance_indicators_sheet`：
  - FY：优先取“主要会计数据和财务指标”章节；
  - Q1/HY/Q3：优先取季度/半年度报告首页指标区；
  - 同比、环比指标若原文已给出则直接取，否则按规则计算；
  - 若某字段无法从原文可靠取得，则允许留空并记录来源缺失。

**典型字段映射示例**：
- `营业总收入` / `一、营业总收入` → `total_operating_revenue`
- `净利润` / `归属于上市公司股东的净利润` →
  - `income_sheet.net_profit` 或 `core_performance_indicators_sheet.net_profit_10k_yuan`（按目标表分别落库）
- `货币资金` → `asset_cash_and_cash_equivalents`
- `应收账款` → `asset_accounts_receivable`
- `存货` → `asset_inventory`
- `交易性金融资产` → `asset_trading_financial_assets`
- `在建工程` → `asset_construction_in_progress`
- `应付账款` → `liability_accounts_payable`
- `合同负债` → `liability_contract_liabilities`
- `短期借款` → `liability_short_term_loans`
- `销售商品、提供劳务收到的现金` → `operating_cf_cash_from_sales`
- `收回投资收到的现金` → `investing_cf_cash_from_investment_recovery`
- `投资支付的现金` → `investing_cf_cash_for_investments`
- `取得借款收到的现金` → `financing_cf_cash_from_borrowing`
- `偿还债务支付的现金` → `financing_cf_cash_for_debt_repayment`

### 5. report_period 表达

**确认结论**：`report_period` 使用完整格式 `{year}{period}`，即 `2023FY / 2024Q1 / 2024HY / 2025Q3`，与附件3及项目上下文保持一致。

**需要强调**：
- 不能在 SQL 层把 `report_period='Q3'` 与 `report_year=2025` 混用作为主约定；主存储格式必须是完整串 `2025Q3`。
- `report_year` 仍单独存数值年，便于过滤和排序。

### 6. NL2SQL：两步式，Schema 固定简化

**选择**：Schema 只有 4 张表，LLM 上下文可以放完整 Schema。

- Step1: LLM 解析意图 → 选表/字段/公司/期间 → JSON
- Step2: LLM 生成 SQL（输入：完整 Schema + Step1 结果 + 示例）

**补充修正**：Step2 生成 SQL 时要统一使用完整 `report_period` 值（如 `2025Q3`），而不是 `Q3 + report_year` 的混合表达，否则会与 ETL 主键约定不一致。

### 7. 多轮对话：上下文拼接

**选择**：维护会话历史，每次新问题拼接上下文后送 LLM。

示例：
```
用户问: "金花股份利润总额是多少"  → LLM 发现缺 report_period → 澄清
用户追问: "2025年第三季度的"     → 拼接上下文 → 生成 SQL
```

此决策暂时成立，但依赖前提是 ETL 阶段对 `report_period` 的标准化必须统一为完整格式（如 `2025Q3`），否则后续 SQL 生成会出现 period 表达不一致。

### 8. 图表生成：matplotlib + 规则优先

**修正建议**：图表选择先用规则实现（趋势→折线，对比→柱状，占比→饼图），Phase 1 不必依赖 LLM 选型。原因同第 1 条：降低波动与外部依赖，优先保证核心链路可交付。

## Risks / Trade-offs

- **PDF 表格格式多变** → 缓解：先用示例 2 家公司调通，正式数据后再扩展规则
- **中文字段名别名多** → 缓解：维护映射字典，未命中字段记录 warning，不把 LLM 当主路径
- **上交所 PDF 报告类型识别** → 缓解：首页标题解析优先，发布日期只做弱兜底
- **核心业绩指标表并非直接等于三大报表** → 缓解：拆分“原值抽取”和“派生指标计算”两层实现
- **勾稽关系校验不通过** → 缓解：记录差异但不阻塞入库，标记为待人工确认
- **LLM 生成 SQL 质量** → 缓解：Schema 小且固定，few-shot 覆盖；执行失败重试 2 次
- **6 小时新数据测试** → 缓解：Pipeline 全自动化，减少非必要在线调用
