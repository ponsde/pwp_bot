## MODIFIED Requirements

### Requirement: build_answer_content detects YoY result columns
`build_answer_content` SHALL detect rows containing `current_value`, `previous_value`, and `yoy_ratio` columns and delegate to YoY-specific formatting logic.

#### Scenario: Normal query result (no YoY columns)
- **WHEN** rows do not contain `yoy_ratio` column
- **THEN** formatting behavior is unchanged (existing "字段名=值" format)

#### Scenario: YoY query result (heuristic path)
- **WHEN** rows contain `current_value`, `previous_value`, `yoy_ratio` (generated by `_build_yoy_sql`)
- **THEN** each row is formatted using YoY natural language template
- **AND** `current_value`/`previous_value` are formatted with appropriate units from `_FIELD_UNITS` lookup using the intent's original field name

#### Scenario: LLM path uses precomputed YoY fields
- **WHEN** LLM-generated SQL uses schema fields like `net_profit_yoy_growth` instead of self-join aliases
- **THEN** existing default formatting applies (no special handling needed)
- **NOTE**: This is acceptable — LLM path results have original field names that map to `_FIELD_LABELS`/`_FIELD_UNITS`
