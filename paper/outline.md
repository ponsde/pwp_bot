## 正文要点（按章展开）

### 第 2 章：任务一 —— 结构化财报数据库构建

**要点**：
- 数据来源：附件 2（1252 份财报 PDF，分 reports-上交所 / reports-深交所
  两种命名规则），附件 1（66 家公司基本信息），附件 3（4 张官方表字段说明）。
- 关键技术：
  1. 双格式元数据识别：上交所文件名携带 `{股票代码}_{报告日期}_{随机标识}`；
     深交所则从 PDF 标题里匹配 `{A股简称}：{年份}年{报告周期}`。通过
     `src/etl/pdf_parser.py` 的 SSE_FILE_RE / SZSE_RE 两条正则统一处理。
  2. 摘要报告识别与跳过：摘要报告内容残缺（无完整资产负债表），
     parser 通过 "摘要" / "报告摘要" 关键字跳过，避免入库噪声。
  3. 表格提取：pdfplumber 扫描前 HEAD_PAGE_LIMIT 页定位主表，再做跨页合并；
     按 `CONFIRM_KEYWORDS` 关键字确认所属报表类型，过滤母公司表等非合并表。
  4. 报告期标准化：中文表述统一为 `2024FY` / `2024Q1` / `2024HY` / `2024Q3`。
  5. 数据校验：资产负债平衡 (≤1%tol)、利润表勾稽、核心指标与利润表交叉一致性。
- 设计权衡：
  - 为什么采用**启发式 + 规则**而非纯 LLM：大量 PDF 解析需要稳定性，LLM 随机
    错误会破坏下游 SQL 查询的正确性。
  - 为什么用 **SQLite** 而不是 MySQL：单文件部署、零依赖、便于提交；
     schema 与附件 3 完全对齐。
  - 为什么把**平衡不等式降级为 warning** 而不是强制 reject：全量测试中，
     约 15% 的 PDF 因单个字段 OCR 失败触发 0.005%-0.1% 失衡，若直接 reject
     会丢失整报告的有效字段；改为记录警告后仍载入，可在后续查询中容忍轻微
     失衡。

### 第 3 章：任务二 —— "智能问数"助手

**架构图要求**：NL → intent extractor → SQL generator → executor → answer/chart renderer

**两步式 Text2SQL**（`src/query/text2sql.py`）：
- `analyze()` 用 `seek_table.md` 提示模板抽取公司、字段、报告期、目标表。
  Heuristic fallback 可在无 LLM 场景下工作（单元测试环境）。
- `generate_sql()` 用 `generate_sql.md` 生成 SQL，再经 SQLite 执行。
- 三层恢复：
  1. SQL 语法错误 → 重生成
  2. 结果越界（空/爆量）→ 触发 `validate_result.md` 校验，再生成
  3. 上述仍失败 → `reflect.md` 重新分析意图

**多轮对话**（`src/query/conversation.py`）：
- `ConversationManager` 维护 history + 槽位(companies, fields, periods)。
- 追问时槽位继承：如"金花股份利润总额是多少"→"2025年第三季度的"时，
  公司、字段从上一轮继承，期间从当前轮补齐。

**意图澄清机制**：
- 检测缺失核心槽位 → 触发 `clarify.md` 模板生成澄清问句（例：
  "请问您要查询哪个报告期的利润总额？"）。

**图表自动选择**（`src/query/chart.py` + `chart_select.md`）：
- 时间序列 → 折线图
- 横向对比 → 柱状图
- 占比 → 饼图
- 单点 → 无图形
- 输出命名：`result/{问题编号}_{顺序号}.jpg`

**实验结果（待填）**：70 道题的作答覆盖率、SQL 可执行率、澄清触发率。

### 第 4 章：任务三 —— 可靠性增强

**知识库**（`src/knowledge/`）：
- 使用 OpenViking 0.3.x 向量库（BAAI/bge-m3 1024 维 embedding）+ VLM 摘要。
- 473 篇研报（303 行业 + 160 个股）批量入库。
- 元数据保留：`paper_path` 用于最终回答引用。

**意图路由**（`research_qa.classify_intent`）：
- `sql`：公司某期数据 → 走 Text2SQL
- `rag`：政策/产品/战略 → 走研报检索
- `hybrid`：数据 + 原因 → SQL + RAG 并行

**多意图拆解**：
- `split_multi_intent` 用 LLM + 正则分句将复合问句拆为可独立回答的子问题。
- 示例："2024 年利润最高的 top10 企业是哪些？这些企业的销售额年同比是多少？
  年同比上涨幅度最大的是哪家企业？" → 3 个子问题。

**归因（references）**：
- 每个 RAG 命中 → `{paper_path, text, paper_image}` 附加到回答。
- 完整链路：用户问句 → 路由决策 → 子任务列表 → 每个子任务的 SQL/检索命中
  → 整合回答。

### 第 5 章：系统实现

- 模块划分：`src/etl/` `src/query/` `src/knowledge/` `src/llm/` `src/prompts/`
- 入口：`pipeline.py`（--task etl/answer/research），`backend/`（FastAPI），
  `web-studio/`（前端 SPA）。
- 测试：pytest，>80 个单元测试；heuristic fallback 使主要路径可无 LLM 运行。
- 部署：Dockerfile 同时启动 vikingbot + FastAPI。

### 第 6 章：结论与展望

- 总结三个任务的核心创新（两步 Text2SQL + 三层恢复、启发式 + LLM 混合 ETL、
  RAG + SQL 混合路由归因）。
- 局限：
  - ETL 对扫描版 PDF 依赖 VLM，耗时与成本较高。
  - 研报 RAG 对 paper_image 支持仍需完善（目前仅返回文本摘录）。
  - 中文数据日期字段混合了"2024 年第三季度 / 2024Q3 / 2024 前三季度"等说法，
    需要更细致的时间实体消歧。
- 展望：
  - 动态 schema：未来不同行业可能需要不同财务表，可引入元 schema 描述层。
  - 增强归因：在每条回答中附加"推理链路"（SQL + 研报摘录 + 计算步骤）。
  - 生产化：完善权限、用户配额、审计日志。
