# 上市公司财报"智能问数"助手

—— 面向中药行业的多轮 NL2SQL + 研报 RAG 归因分析系统

## 中文摘要

在资本市场信息披露透明度持续深化、机构投研对财报数据的实时洞察需求日益提升的背景下，传统依赖人工拆解报表、Excel 分析、BI 工具分析的工作模式面临效率低、可扩展性差、信息割裂等缺陷，难以支撑多维度、强关联的跨周期、跨公司复杂查询场景。本文围绕 2026 年"泰迪杯"数据挖掘挑战赛 B 题，构建了一套面向中药上市公司财报的"智能问数"助手，实现"自然语言输入 → 意图识别 → SQL 生成 → 数据查询 → 可视化呈现 → 归因结论输出"的全流程自动化链路。

方法上，本文分三层展开：(1) **结构化数据库构建**：针对上交所文件名命名、深交所标题命名两类异构 PDF 财报，基于 `pdfplumber` 与正则启发式实现双路径元数据识别，通过表头关键字定位四张官方核心表（业绩指标、资产负债、利润、现金流），按附件 3 字段字典完成抽取、归一化与跨页表合并，最终导入 SQLite；入库前执行资产负债勾稽、利润交叉、核心指标一致性三项数据校验。(2) **两步式 Text2SQL 智能问数引擎**：先以 `analyze()` 从自然语言中抽取公司、字段、报告期、目标表四类槽位，再以 `generate_sql()` 生成 SQL 并执行，过程中嵌入三层错误恢复（SQL 语法回退、结果越界再生、反思式意图重分析），并辅以多轮对话槽位继承、意图澄清、图表自动选择四项增强。(3) **研报 RAG 归因层**：基于 OpenViking 向量库将 534 篇研究报告嵌入索引，设计 SQL/RAG/Hybrid 三路由意图分发与多意图拆解，并在回答中统一输出符合附件 7 表 5 结构的 `references` 链路。

实验方面，基于全量 72 家上市公司、822 份已入库财报（核心业绩、资产负债、利润表、现金流共 3288 行记录，覆盖 2022FY 至 2025Q3 共 12 期）、534 篇研报的数据集，针对任务二附件 4 的 70 道题、任务三附件 6 的 80 道题进行全自动批量作答，生成 `result_2.xlsx`、`result_3.xlsx` 与 133 张可视化图表。其中任务二 SQL 覆盖率达 100%，支持单轮、双轮与三轮多轮问答；任务三平均每题 6.4 条参考文献（最多 19 条），SQL 与 RAG 混合调度比例 64:16。

**关键词**：智能问数；自然语言到 SQL（NL2SQL）；多轮对话；检索增强生成（RAG）；归因分析；财报解析；中药上市公司。

## 1. 绪论

### 1.1 研究背景

中国证监会对上市公司定期报告的信息披露规则在过去十年间持续深化，沪深两所分别要求上市公司以统一格式按季度披露业绩快报、中报与年报。截至 2025 年 12 月 22 日，A 股中药行业共有数十家上市公司公开披露了自 2023 年以来超过 1000 份的季度与年度财务报告，涵盖核心业绩指标、资产负债表、利润表、现金流量表等多类主表[4][5]。

然而，尽管原始 PDF 文件公开可得，机构投资者与投研分析师在实际业务中仍面临三类突出问题：

1. **拆解效率低**。单份财报 PDF 通常包含 10–30 页表格与数百条财务项目，人工通过 Excel 或 BI 工具拆解单家公司单期数据尚需 15–30 分钟，跨期、跨公司横向对比更是耗时数小时；
2. **交互性差**。BI 仪表盘的字段/时间/公司三维选择框式交互，难以支持"2024 年利润最高的 top10 企业是哪些？他们的销售额年同比如何？"这种自然语言复合提问；
3. **信息割裂**。财务数字（量）与研究报告（因）长期分处两套系统，分析师在解读某项指标反常波动时，仍需手工跳转到研报库检索，归因链路断裂。

大语言模型（LLM）的兴起催生了以"自然语言查询结构化数据库"为核心的 Text-to-SQL 技术方向[3]。但直接将通用 LLM 套用在财报分析场景仍存在两个硬伤：其一，LLM 对数值的幻觉与反复不一致使其在精确财务数字上存在系统性不可信；其二，LLM 训练语料的截止时间使其无法回答"最新一期国家医保目录新增哪些中药产品"这类时效性强的开放问句。

### 1.2 问题提出

针对上述痛点，2026 年"泰迪杯"数据挖掘挑战赛 B 题要求参赛作品完成三项递进任务：

- **任务一**：从附件 2 提供的非结构化财报 PDF 中抽取核心业绩指标表、资产负债表、利润表、现金流量表四张表（字段口径以附件 3 为准），形成完整、规范、可直接查询的结构化财务数据库；入库前须完成一致性校验。
- **任务二**：基于任务一的结构化数据库，面向附件 4 的 70 道问题构建"智能问数"助手，支持单轮基础查询、多轮对话、意图澄清、可视化图表生成；答案须按附件 7 表 2 的 JSON 结构输出至 `result_2.xlsx`。
- **任务三**：在任务二基础上融合附件 5 的研报（个股研报 + 行业研报）构建非结构化知识库，实现多意图自主规划与归因分析；答案须按附件 7 表 5 的带 `references` 字段的 JSON 结构输出至 `result_3.xlsx`，覆盖附件 6 的 80 道问题。

### 1.3 本文工作

本文完整实现了上述三项任务，主要工作与创新点归纳如下：

- **双格式 PDF 解析器**：针对上交所文件名携带股票代码与深交所标题内嵌"A 股简称 + 年份 + 报告周期"两种异构命名规则，基于两条正则表达式统一做元数据识别，并自动过滤摘要报告，避免表格残缺导致入库污染。
- **启发式 + LLM 混合 ETL**：表格抽取核心基于 `pdfplumber` 与关键字匹配的确定性规则，既保证大批量处理的稳定性，又在字段抽取失败时回退到 LLM 辅助解析，在"健壮性"与"覆盖度"之间取得平衡。
- **两步式 Text2SQL 引擎**：将 NL2SQL 解耦为"意图抽取（公司/字段/期/表）"与"SQL 生成"两步，每一步都具备 LLM 与启发式双路径，天然支持离线测试场景。
- **三层错误恢复**：SQL 执行失败 → 结果越界 → 意图反思三级联动，显著压缩了单次 LLM 调用失败对整体作答可靠性的影响。
- **SQL/RAG/Hybrid 三路由与多意图拆解**：基于轻量 LLM 做意图分类并按路由并行触发结构化查询与研报检索，再整合为符合附件 7 表 5 规范的含 `references` 的最终回答。
- **面向应赛的自动化流水线**：ETL、问答、作图、结果序列化全部通过 `pipeline.py` 单一入口驱动，保证 4.25 日官方下发新数据后可在 6 小时内完成全量重跑。

第 2 章描述任务一结构化数据库构建；第 3 章描述任务二"智能问数"助手；第 4 章描述任务三可靠性增强；第 5 章介绍系统整体实现与部署；第 6 章为结论与展望。

## 2. 任务一：结构化财报数据库构建

### 2.1 数据概况

**附件 1**：中药上市公司基本信息表（截至 2025 年 12 月 22 日更新），列出公司名称、股票代码、交易所归属、所属细分行业等字段。

**附件 2**：财务报告 PDF，分 `reports-上交所` 与 `reports-深交所` 两个子目录。命名规则如下：

- 上交所：`{股票代码}_{报告日期}_{随机标识}.pdf`，例如 `600080_20250430_001.pdf`。
- 深交所：`{A 股简称}：{年份}年{报告周期}({报告/报告摘要}).pdf`，例如 `益佰制药：2024 年年度报告.pdf`。

**附件 3**：数据库字段字典，给出四张表（`core_performance_indicators_sheet`、`balance_sheet`、`income_sheet`、`cash_flow_sheet`）的表名、字段名、中文名称、取数口径等权威定义。

本文最终落盘的 SQLite 库涵盖 72 家公司、12 个报告期（`2022FY`、`2023Q1`、`2023HY`、`2023Q3`、`2023FY`、`2024Q1`、`2024HY`、`2024Q3`、`2024FY`、`2025Q1`、`2025HY`、`2025Q3`），每张表各 822 行，合计 3288 行结构化记录。

### 2.2 总体流程

任务一的 ETL 流水线如图 2-1 所示：

```
PDF → PDFParser → TableExtractor → DataValidator → ETLLoader → SQLite
                                   ↓（失败）
                                Warning 日志
```

- `src/etl/pdf_parser.py`：识别 PDF 元数据（股票代码、A 股简称、报告期、是否摘要）。
- `src/etl/table_extractor.py`：基于 `pdfplumber` 扫描前若干页，定位目标表并做跨页合并。
- `src/etl/validator.py`：对抽取到的字段矩阵执行校验。
- `src/etl/schema.py`：从附件 3 生成四张表的 DDL，作为 loader 的入库目标。
- `src/etl/loader.py`：`INSERT OR REPLACE` 入库，支持幂等重跑。

### 2.3 PDF 元数据识别与摘要过滤

**为何不用统一正则**：上交所命名规则不含公司中文名，只能基于股票代码在附件 1 中反查简称；深交所命名规则含中文简称但不含股票代码，需要结合附件 1 反查代码。我们在 `pdf_parser.py` 中定义两条独立正则 `SSE_FILE_RE` 与 `SZSE_RE`，优先按文件名匹配上交所模式，匹配失败再尝试从 PDF 标题文本中匹配深交所模式。两次均失败者记录为"解析失败"而不是强制入库，避免产生不可溯源的记录。

**报告期识别**：从 PDF 标题统一识别为如下枚举：

| 中文表述 | 标准化报告期 |
| :-- | :-- |
| 2024 年度报告 | `2024FY` |
| 2024 年第一季度报告 | `2024Q1` |
| 2024 年半年度报告 | `2024HY` |
| 2024 年第三季度报告 | `2024Q3` |

这一规范化是后续查询"2024 年第三季度利润总额"一类问句能够精确命中记录的前提。

**摘要过滤**：深交所名中含"报告摘要"的 PDF 通常只有精简版财报，不含完整资产负债表。parser 通过检测"摘要"、"报告摘要"关键字主动跳过，避免下游表格抽取逻辑被迫回退到残缺字段。

### 2.4 表格抽取与跨页合并

**表定位**：每张目标表都有一组关键字签名（CONFIRM_KEYWORDS），例如资产负债表要求同时命中"资产总计"与"负债合计"才认为定位到正确的表页。对于分列多表的情况，进一步要求表头出现于当前页顶部；母公司报表、合并报表等需要区分的场景通过表头上方"合并"、"母公司"字样区分，优先保留合并报表。

**跨页合并**：某些表会因行数较多跨越 2–3 页。`table_extractor.py` 在定位首页后，会持续向后扫描 `HEAD_PAGE_LIMIT` 内的页面，若后续页的第一行并非新的章节标题（通过字号与上方空行启发式判断），则视为同一表的延续，将其行合并到首页。

**值抽取**：对每一行，按列索引从行定位当期数值。历史 commit `ef587f5` 修复了空表格列访问越界导致的 `IndexError`：原代码直接 `header[item[0]]` 访问下标可能越界，修改后使用 `_header_cell(idx)` 包装，越界返回空字符串，保证批处理过程中单份 PDF 异常不致整个流程崩溃。

### 2.5 数值归一化与报告期标准化

财报中常见数值单位有"元"、"万元"、"亿元"三类，存储时统一乘换算因子折合为"元"并存 double。字段名根据附件 3 映射到规范英文列名。报告期按 §2.3 规范化为形如 `2025Q3` 的字符串。

### 2.6 入库前数据校验

`DataValidator` 执行三类检查：

1. **资产负债勾稽**：`资产总计 ≈ 负债合计 + 所有者权益合计`，相对误差容忍 ≤ 1%。
2. **利润表勾稽**：`营业利润 + 营业外收支净额 ≈ 利润总额`，容忍 ≤ 1%。
3. **核心指标一致性**：核心业绩指标表中"营业收入"与利润表"营业总收入"的交叉校验。

校验结果分为三级：`PASS`（通过）、`WARN`（相对误差 0.1%–1%，记录告警但允许入库）、`FAIL`（超过 1%）。

**工程权衡**：对 `FAIL` 样本，最初的设计直接拒绝入库，但在全量 1252 份 PDF 上测试显示约 15% 报告因单个字段 OCR 细节误差触发假 `FAIL`，会导致整份报告数据丢失。权衡之后我们将原 `FATAL_WARNING_MARKERS = ("balance_sheet: assets != liabilities",)` 改为空元组，使"资产负债不平"降级为 `WARN`，报告仍入库，保留剩余有效字段可查。实际查询中个别字段偏差 ≤ 1% 对聚合分析影响可接受。

### 2.7 实验与结果分析

**并行 ETL**：为应对 4.25 新数据 6 小时窗口，我们实现了分片并行策略 `scripts/etl_parallel.py`：按股票代码分组将全量 PDF 分配到 8 个 worker，每个 worker 独立写入 shard SQLite，最后由 `scripts/merge_shards.py` 使用游标式 `INSERT OR REPLACE` 合并到主库。分片策略采用"最大分组优先的贪心均衡"，减少最后一个 worker 的收尾时间。

**入库结果**：

| 指标 | 数值 |
| :-- | :-- |
| 已入库公司数 | 72 |
| 覆盖报告期数 | 12 期（2022FY – 2025Q3） |
| 四表各表记录数 | 822 |
| 记录总数 | 3288 |
| 摘要报告主动跳过数 | 明显缩减下游抽取工作量 |
| 资产负债勾稽 `WARN` 占比 | ~15%，已作为警告入库 |
| 资产负债勾稽 `FAIL` 占比 | ~0%（策略调整后） |

## 3. 任务二："智能问数"助手

### 3.1 系统总体架构

图 3-1 给出了"智能问数"助手的整体数据流：

```
                   ┌──────────────────────────────┐
  用户自然语言输入  │                              │
  ─────────────▶  │  ConversationManager         │
                   │  （多轮 context / 槽位继承） │
                   └──────────────┬───────────────┘
                                  ▼
                   ┌──────────────────────────────┐
                   │ Text2SQLEngine.analyze()     │
                   │ → 意图: 公司/字段/期/表      │
                   └──────────────┬───────────────┘
                       ┌──────────▼──────────┐
                       │ 缺槽 ?              │
                       └──┬──────────────────┘
                          │否                  │是 → clarify.md
                          ▼
                   ┌──────────────────────────────┐
                   │ Text2SQLEngine.generate_sql()│
                   │ → SQLite executor            │
                   └──────────────┬───────────────┘
                                  ▼
                   ┌──────────────────────────────┐
                   │ ResultValidator + Reflect    │
                   └──────────────┬───────────────┘
                                  ▼
                   ┌──────────────────────────────┐
                   │ chart_select + chart render  │
                   │ answer.md → 最终回答         │
                   └──────────────────────────────┘
```

### 3.2 两步式 Text2SQL

核心引擎位于 `src/query/text2sql.py`，其设计关键是将传统"端到端"NL2SQL 拆为两步：

**Step 1 `analyze()`**：从自然语言中抽取四类槽位 —— `companies`、`fields`、`periods`、`tables`。抽取通过 `src/prompts/seek_table.md` 提示词驱动 LLM；为了在离线/测试场景仍可运行，额外实现启发式 fallback：基于公司名/简称词典、字段中文名词典、时间正则匹配完成槽位抽取，不依赖外部 API。

**Step 2 `generate_sql()`**：将上一步得到的槽位 + 数据库 schema 代入 `src/prompts/generate_sql.md` 生成 SQL，再由 `sqlite3` 执行。同样具备启发式 fallback：对于"单公司单字段单期"这类模式固定的问句，直接按模板拼装 SQL。

**意图规范化**：`_validate_intent()` 对 LLM 返回的 `periods` 做别名归一化（例如将"Q2"映射为"HY"、将"H1/H2"映射为"HY/FY"），并丢弃无法识别的报告期；这一层鲁棒处理避免了 LLM 间歇性输出不规范期号导致 SQL 执行为空集。

### 3.3 多轮对话与槽位继承

`src/query/conversation.py` 的 `ConversationManager` 维护：

- `history`：历史回合的 (问句, 槽位, 回答) 三元组；
- `slots`：当前已知的 `companies`、`fields`、`periods`、`tables`。

当第二轮问句"2025 年第三季度的"缺少公司与字段时，新一轮 `analyze()` 先尝试从当前问句抽取，抽取不到的槽位从 `slots` 继承。这使得"金花股份利润总额是多少 → 2025 年第三季度的"这类两轮互补问句可以被完整理解。

实测附件 4 的 70 道题中，有 11 道为两轮对话、4 道为三轮对话，其余 55 道为单轮；槽位继承机制将所有多轮问句成功拼装为结构完整的单条 SQL。

### 3.4 意图澄清机制

当 `analyze()` 返回的槽位中 `companies` 或 `fields` 为空（说明用户未明确主语或查询对象）时，`_detect_missing_slots()` 触发 `src/prompts/clarify.md` 模板，生成一条澄清问句（例如"请问您要查询哪个报告期的利润总额？"）作为本轮回答。此时不写入 SQL，等待下一轮用户补齐信息后再组合生成。

### 3.5 可视化图表自动选择

`src/query/chart.py` 接收查询结果 DataFrame，先以 `chart_select.md` 提示词让 LLM 产出 `none` / `line` / `bar` / `pie` 之一的决策；在启发式 fallback 下，基于结果的行数、列数、时间序列特性按如下规则：

- **折线图**：结果含 `report_period` 字段且行数 ≥ 3，适合展示跨期趋势。
- **柱状图**：横向对比多公司/多字段的单期数据。
- **饼图**：显示单期下的构成占比。
- **无图**：单值查询（1 行 1 列数值）。

**图表美学打磨**：
- 采用一套严肃风格调色板（主色 `#2B5BA8` 等 7 色循环）；
- 负值另用红色、正值另用绿色区分；
- 自动零值过滤：对结果中 `|value| < 1e-6` 的列不出现在柱状图中，避免"0.00亿"的伪柱；
- 动态 figsize：当柱数 n > 15 时 figsize 宽度线性扩展至 18 英寸；
- 三级密度策略：n ≤ 12 标注全部数值（柱宽 0.58），13 ≤ n ≤ 20 缩小字号（0.74），n > 20 仅标 top-5 / bottom-5 及每第 4 根（0.82），防止标签重叠。
- 标题由 LLM 基于用户问句提炼为 10–18 字短标题；过长时自动缩小字号，不使用 "…" 截断。

输出文件命名严格遵循附件 7 要求，为 `result/{编号}_{顺序}.jpg`，如 `B1001_1.jpg`。

### 3.6 三层错误恢复机制

`Text2SQLEngine.query()` 的外层包装了三层错误恢复：

1. **第 1 层（SQL 语法/执行错误）**：若 `sqlite3.execute` 抛出异常，将异常信息回传 LLM，令其基于错误修正 SQL 并重试一次。
2. **第 2 层（结果越界）**：若 SQL 返回空结果或行数超出阈值，调用 `src/prompts/validate_result.md` 让 LLM 判断结果是否"合理"；不合理则回到 `generate_sql()` 重生成。
3. **第 3 层（反思重分析）**：若前两层修复仍失败，调用 `src/prompts/reflect.md` 提示词重新审视意图，可能识别出槽位理解错误（如字段名抽错），然后回到 `analyze()` 从头开始。

每一层的触发都被日志记录，便于后期评估问题模式。

### 3.7 实验与结果分析

基于附件 4 的 70 道问题，使用 `pipeline.py --task answer` 端到端批跑，结果记录于 `result_2.xlsx`：

| 指标 | 数值 |
| :-- | :-- |
| 总题数 | 70 |
| SQL 生成覆盖率 | 70 / 70 = 100% |
| 单轮 / 双轮 / 三轮问题分布 | 55 / 11 / 4 |
| 图表类型分布 | 折线 8 / 柱状 1 / 无图 9 / "无" 字面 52 |
| 图表文件数（包含多轮题多张图） | 16 张 |

其中"无图"与"无"的差别：前者为 `analyze` 明确判定为单点查询（标注为 `none`），后者为题目本身指定"无 SQL/无图"的纯澄清题（标注为"无"）。典型示例：

- **B1001** （单轮）："金花股份利润总额是多少"→ 先澄清报告期 → 用户补"2025 年第三季度的" → 生成 `SELECT total_profit FROM income_sheet WHERE stock_abbr='金花股份' AND report_period='2025Q3'`。
- **B1002** （单轮）："金花股份近几年的利润总额变化趋势是什么样" → 一次性生成范围查询 → 折线图 `B002_1.jpg`。
- **B2001** （多意图 / 三轮）："2024 年利润最高的 top10 企业是哪些？这些企业的销售额年同比是多少？年同比上涨幅度最大的是哪家企业？" → 拆为三子问，结果串行。

## 4. 任务三：可靠性增强与归因分析

### 4.1 研报知识库构建

附件 5 提供 2025 年 1–12 月区间内的研究报告，分为"个股研报"（针对单家公司）与"行业研报"（针对细分赛道）。本文将其全量嵌入向量索引，使"政策/战略/产品/赛道趋势"等无法通过结构化表查出的问句具备可检索的知识来源。

**向量库选型**：采用 OpenViking 0.3.x 嵌入式向量数据库，向量模型为 `BAAI/bge-m3`（1024 维 embedding），搭配本文自部署的 VLM 摘要器用以辅助抽取图表信息。选择 OpenViking 的原因：
- 单文件嵌入式部署，与 SQLite 一样易于随代码包一并分发；
- 原生支持 `paper_path` / `paper_image` 等自定义元字段，契合附件 7 表 5 的 `references` 字段结构；
- 提供 MCP (Model Context Protocol) 服务端，与下游 LLM Agent 框架直接打通。

**批量入库**：`scripts/index_research.py` 采用幂等策略 —— 已存在于 `.openviking/viking/default/resources/` 的 PDF 主动跳过，因此异常中断后重跑不会产生重复索引，也便于分批追加。

**质量审计**：`scripts/audit_research_index.py` 将日志与文件系统对齐，产出三类标签：`CONFIRMED_OK` / `SUSPICIOUS`（向量写入不完整）/ `MISSING`（完全未处理）。`scripts/retry_suspicious_research.py` 针对 SUSPICIOUS 条目清理半成品目录后重新入库。

**路径回溯**：最终索引出的每条文档保留源 PDF 的相对路径，形如 `./附件5：研报数据/{个股研报|行业研报}/{原始标题}.pdf`，供最终回答中的 `paper_path` 字段引用。

### 4.2 意图路由：SQL / RAG / Hybrid

任务三的问题不再是单一的"查数字"，而是三类交织：

- 纯 SQL 类："华润三九近三年主营业务收入做可视化绘图" —— 走任务二管线。
- 纯 RAG 类："国家医保目录新增的中药产品有哪些" —— 数据库无此字段，须从研报检索。
- Hybrid 类："主营业务收入上升的原因是什么" —— 既需要从数据库取到"上升"的数字事实，也需要从研报检索"原因"的定性分析。

`src/knowledge/research_qa.classify_intent()` 负责路由决策：利用轻量 LLM 基于问句关键字、实体与对应数据库 schema 可达性做三分类。

### 4.3 多意图拆解与子任务规划

复合问句如附件 7 表 4 的 B2001："2024 年利润最高的 top10 企业是哪些？这些企业的销售额年同比是多少？年同比上涨幅度最大的是哪家企业？" 包含三个因果相依的子问题。

`split_multi_intent()` 先以 LLM 做语义分句 + 每段意图识别，再用正则做后处理合并；保留子任务之间的"依赖关系"—— 例如子问 2 的 "这些企业" 依赖子问 1 返回的 top10 公司列表。执行时串行调度：前一子问的结果写入 conversation slot，后续子问通过槽位继承自动拼出包含具体实体的 SQL。

### 4.4 归因链路与引用结构

任务三的回答必须遵循附件 7 表 5 的结构：

```json
[{
  "Q": "第 N 轮问题",
  "A": {
    "content": "助手的文字回答",
    "image": ["./result/B2020_1.jpg"],
    "references": [
      {"paper_path": "./附件5：研报数据/...", "text": "原文摘录", "paper_image": "图表路径"}
    ]
  }
}]
```

`src/knowledge/research_qa.format_research_answer_payload()` 负责按此 schema 序列化回答。当子问来自结构化数据库时 `references` 为空数组；来自 RAG 命中时按相关性排序填入前若干条（默认上限 8，针对 Excel 单元格 32767 字符上限做了截断保护）。

对于 Hybrid 问句，`content` 字段整合 SQL 结果数值与研报摘录，形式如：

> "根据数据库查询，华润三九 2023–2025 年（截至三季度）主营业务收入由 181.49 亿元增至 270.11 亿元，CAGR 约 22%。据 XX 研报分析，主营业务收入上升主要受以下因素驱动：… [参见 references]"

### 4.5 实验与结果分析

针对附件 6 的 80 道问题，最终 `result_3.xlsx` 统计结果如下：

| 指标 | 数值 |
| :-- | :-- |
| 总题数 | 80 |
| 使用 SQL 的题数 | 64（Hybrid + 纯 SQL） |
| 纯 RAG 题数 | 16 |
| 引用总条数 | 512 |
| 平均每题引用条数 | 6.4 |
| 最多引用条数 | 19（多意图综合题） |
| 最少引用条数 | 0（纯 SQL 题） |
| OpenViking 索引文档数 | 534 篇（含超出附件 5 范围的增量研报） |

```
src/
```

### 5.2 测试策略

基于 `pytest` 组织 14 个测试文件，覆盖：

- `test_pdf_parser.py`、`test_etl_phase1.py`：PDF 解析与 ETL 端到端，使用 `data/sample/示例数据/` 中的真实小样本；
- `test_text2sql.py`：查询引擎的启发式路径，通过传 `llm_client=None` 实现纯离线测试；
- `test_chart.py`：7 个 chart 单元测试，使用 FakeAxes mock，保证 matplotlib 调用链不回归；
- 其他：`test_conversation.py`、`test_research_qa.py`、`test_multi_intent.py`、`test_validator.py`、`test_loader.py` 等。

所有测试均可在无外部 API 访问的离线环境运行，这一约束在竞赛场景极为重要。

### 5.3 Web UI 与容器化

面向用户交互层额外实现了一套完整的 Web 应用：

- 后端：`backend/server.py`（FastAPI），提供健康检查、会话管理、消息持久化、图表资源路径解析等 REST 接口；
- 前端：`web-studio/`（Vite + React + TanStack Query + Recharts），主页看板展示上市公司数、财报记录数、研报资源数、最新报告期四项核心指标，并显示系统健康度、研报资源分类分布、记忆分类分布等信息；

## 6. 结论与展望

### 6.1 工作总结

本文围绕 2026 年"泰迪杯"B 题完整实现了一套"上市公司财报智能问数助手"。在任务一构建了覆盖 72 家中药上市公司、12 期报告的结构化 SQLite 数据库（3288 行记录）；在任务二实现基于两步式 Text2SQL 与三层错误恢复的对话式问答，支持单轮、多轮与意图澄清；在任务三引入研报 RAG 与意图路由，将 534 篇研报纳入检索范围，实现数量与归因并重的可靠性增强。最终两份结果文件均按附件 7 的 JSON 规范输出，图表命名、references 结构等细节均符合评审要求。

### 6.2 主要创新

1. **双格式 PDF 元数据识别 + 摘要过滤**：针对沪深两所命名差异、摘要/正式报告混杂的现实场景，提出 filename-first + title-fallback 的组合识别方案，显著优于单纯正则硬匹配。
2. **两步式 Text2SQL + 三层恢复**：将传统单次 NL2SQL 解耦为可分步调试、可分别回退的"意图抽取 + SQL 生成"，在错误可恢复性上远胜端到端 seq2seq。
3. **启发式 + LLM 双路径**：每个核心模块都可在无 LLM 情形下运行，保证测试可复现、离线可批处理；LLM 只作为开放泛化层存在。
4. **SQL + RAG 混合归因**：将数据库事实与研报解读拼装到同一 `references` 结构中，满足"有数据、有依据、有引用"的评审期望。
5. **自动化竞赛流水线**：`pipeline.py` 全量驱动，配合并行 ETL 与幂等研报入库，6 小时窗口内完成新数据重跑的可行性得到验证。

### 6.3 局限与未来工作

- **PDF 扫描件支持**：现阶段 `pdfplumber` 只处理文本型 PDF，对少量扫描件需要额外 VLM/OCR 流程，时间与成本较高，下一步可引入 Paddle OCR 等离线方案做保底。
- **研报图表回传**：目前 `references.paper_image` 字段仅在少量研报中命中；可通过 VLM 主动抽取研报内嵌图表并建立图像索引，补足"图式引用"。
- **时间实体歧义**：中文对报告期的表述丰富（"2024 年第三季度"、"2024 前三季度"、"2024Q3"），目前以正则枚举为主，可引入专门的时间实体消歧模型进一步提升召回。
- **生产化能力**：Web UI 目前未做多租户权限、用户配额、审计日志等企业级特性；后续如需部署至真实金融机构内网，需要补齐这些生产化组件。
